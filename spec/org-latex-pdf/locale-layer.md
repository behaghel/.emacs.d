# Locale Layer

## Context

Locale behavior is a separate architectural layer so that classes can ship English first without hard-coding English into their implementation model.

## Supported v1 Locales

- English: `en`
- French: `fr`

No other locale codes are in scope for v1.

## Rollout Rule

- The first shipped locale is English.
- French is still modeled from day one as a first-class locale profile.
- “Modeled from day one” means locale-aware strings, package selection, and validation paths must already have a place in the design even if the first green slice only exercises English.

## Locale Selection Contract

- Locale is selected with `#+LANGUAGE:`.
- If `#+LANGUAGE:` is absent, the export path uses the English locale profile by default for v1.
- If `#+LANGUAGE:` is present with an unsupported value, export should fail explicitly rather than silently degrade to another locale.

## Locale-Owned Surfaces

The locale layer owns the following output behavior:

- LaTeX language package or preamble configuration.
- Hyphenation rules.
- Locale-generated labels such as table of contents, figure, and table labels.
- Date formatting and month names when generated by the export pipeline.
- Locale-owned callout labels or other generated chrome text, if a class emits such text.
- Locale-sensitive spacing or typography rules such as French punctuation spacing where the chosen engine supports it.

The locale layer does not own authored prose, brand tone, or class-specific emphasis.

## Package and Engine Policy

- The locale layer may be implemented through `org-latex-packages-alist`, `LATEX_HEADER`, `LATEX_HEADER_EXTRA`, or related class-level preamble wiring.
- This spec intentionally does not lock v1 to either Babel or Polyglossia yet.
- The accepted implementation must support English and French without requiring a derived backend by default.
- If engine constraints later force a different language-package strategy, that change should remain inside the locale layer rather than leaking into class specs.

## Class Interaction Rules

- Class specs may request locale-owned chrome such as localized labels.
- Class specs may not redefine locale selection rules.
- A class may stage which localized surfaces are exercised in the first slice, but it may not collapse the locale boundary back into hard-coded English.

## Validation Expectations

- The English slice must prove that locale selection exists even if only the `en` profile is executed end to end first.
- Later French specimens must verify both generated labels and representative body-text extraction for accented characters and spacing-sensitive punctuation.

## Non-Goals

- This file does not define full translation copy for every future class.
- This file does not decide the final LaTeX engine before implementation evidence exists.
- This file does not require French to be fully rendered in slice 1.

## Acceptance Signals

- English-first shipment and day-one locale separation are both stated explicitly.
- Missing locale keywords and unsupported locale values have unambiguous behavior.
- Locale responsibilities are separated from class styling and semantic authoring.
